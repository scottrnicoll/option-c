2. Glossary — Math & Game Mechanics

These terms describe how math is organized and turned into gameplay.

TermDefinition
Game MechanicThe core gameplay pattern tied to a math domain. There are 19 mechanics (e.g. "Balance & Equalize", "Split & Share"). Each has a stick figure animation that demonstrates the mechanic. The mechanic determines HOW the player interacts with the math.
Math RoleAuto-filled from the Game Mechanic. Describes how math fits into the game (e.g. "Add and remove to make both sides match"). The learner doesn't choose this — it comes from the mechanic.
Game EngineThe pre-built game code for a mechanic. Handles all game logic (rounds, scoring, win/lose, difficulty). There are 19 engines (one per mechanic), each built on the shared Phaser base (Boot, Victory, GameOver, HUD, game juice). The engine is the MACHINE that runs the game. Each engine file contains ONLY the gameplay code — everything else comes from the shared base. Engines are tested and guaranteed to work.
Game OptionA specific gameplay mode within a Game Engine. Each engine has 3 game options (57 total). Each option is a genuinely different way to play the same math — not just a difficulty tweak. Example: "Balance & Equalize" has Free Balance, Mystery Side, and Chain Scales. The learner picks one when building their game. Each game option card shows: the option name + a short description of what the player does. In code, game options use string IDs like "free-collect", "conveyor-belt", "split-the-loot" — replacing the old "classic"/"timed"/"challenge" variant system.
Theme ConfigA JSON object generated by AI from the learner's Circuit Board Builder picks. Contains names, colors, messages, and sprite selections that skin the Game Engine. The config is the PAINT on the machine. AI generates it in ~2 seconds.
Sprite Library30 SVG assets used in games: 10 characters (pirate, robot, astronaut, knight, chef, diver, ghost, ninja, wizard, explorer), 10 items (coin, gem, treasure chest, crystal, potion, fruit, star, shell, mushroom, key), 10 backgrounds (underwater, space, forest, castle, kitchen, cave, city, volcano, arctic, jungle). Learner picks directly from grids in the Circuit Board Builder. Built by Mike.
Sprite PickerThe sprite grids inside the Circuit Board Builder. Learner clicks a thumbnail to select a background, character, or item from the library. No separate screen — sprites are picked inline as part of the builder. Sprite library built by Mike.
Round DataAI-generated math problems tailored to the specific Common Core standard. Each round has: a prompt (what the player sees), a target (correct answer), items (clickable numbers including answer + distractors), and a hint. Generated by Haiku at build time. Built by Mike.
Phaser EngineA next-generation game engine using the Phaser.js game framework (vs our original simple HTML engines). Currently only Collect & Manage uses Phaser. Supports sprite rendering, physics, and richer gameplay. Built by Mike.

7. All 19 Game Engines

Every Game Mechanic has a pre-built Game Engine with 3 game options each (57 total). The engine handles all game logic. AI only generates the Theme Config (names, colors, messages) — not game code. Generation takes ~3 seconds. See the full Game Options table below for all 57 gameplay modes.

Engine Features (all 19 engines share these)

  • 5 rounds with progressive difficulty
  • Randomized values every playthrough
  • Game juice: particle bursts (triple on victory), screen shake, combo counter, score pop-ups
  • SVG-drawn characters (no emoji)
  • Intro screen explaining how to play
  • Help button (?) with math explanation + valid/invalid examples
  • gameWin() / gameLose() protocol for parent app integration
  • Fail tracking: 3 wrong answers = game over (triggers Hint Card)
  • Mouse, touch, and keyboard input with smooth hover/drag feedback
  • Consistent visual style with Space Grotesk (titles) + Lexend (body) fonts
  • Theme-specific background colors generated by AI to match learner's chosen theme

Engine Architecture: Shared Base vs. Per-Engine Code

Every Phaser engine is split into two layers: a shared base (identical across all 19 engines) and per-engine code (unique to each mechanic and its 3 game options). This separation means each engine file focuses purely on gameplay — all infrastructure is handled once.

Shared Phaser Base (base-phaser-template.ts)

The shared base is a function that wraps any engine's gameplay scenes into a complete, playable HTML file. Every game gets these for free:

  • BootScene — loading bar, preloads character/item/background sprites from the sprite library
  • VictoryScene — character celebration animation, particle fireworks, score display, dare text, sends game_win postMessage to parent app
  • GameOverScene — "game over" feedback with wrong-answer count, sends game_lose postMessage (triggers Hint Card)
  • HUD overlay — game title (Space Grotesk), score counter, 5 round dots, 3 life hearts, help button (?)
  • Game juice library — burstParticles(), screenShake(), comboCounter(), scorePop() available to all engines
  • Fonts — Space Grotesk (titles) + Lexend (body) loaded via Google Fonts CDN
  • Phaser config — 800×600, Scale.FIT, CENTER_BOTH, arcade physics
  • postMessage protocol — game_win and game_lose for parent app integration
  • Tutorial overlay — "How to play" screen shown before first round, with text provided by the engine

Per-Engine Code (what each engine file provides)

Each engine file (e.g. collect-manage-phaser.ts) contains ONLY the gameplay that makes that mechanic unique. It provides four things:

ComponentWhat it isExample (Collect & Manage)
GameScene class(es) The Phaser Scene(s) containing the core interaction loop — what the player sees, clicks, drags, or types each round. Each game option can be a separate scene class, or options can branch within one scene if they share enough logic. This is where the game FEELS different between options. Free Collect: items appear on screen, click to add to total, hit exact sum.
Conveyor Belt: items scroll left-to-right on a belt, tap to grab before they fall off the edge.
Split the Loot: two bins at bottom, drag items into either bin, each bin has its own target sum.
Round generation logic A function that produces the numbers, positions, and targets for each round. Takes the current round number (1–5) and difficulty curve, and returns: the target value(s), the item values to display, and their positions/timing. Must guarantee at least one valid solution exists. When AI-generated RoundData is available (math-standard-aware problems), the engine uses those instead of its own random generation. Round 1: target=12, 5 items (values 3,4,8,5,7), subset {5,7} sums to 12.
Round 5: target=48, 9 items with tighter margins and more distractors.
Intro text A short, plain-language explanation shown on the tutorial overlay before the game starts. Must tell the player: (1) what they'll see on screen, (2) what they need to do, and (3) what counts as right vs. wrong. Written for the grade level of the learner. Free Collect: "Items appear with numbers on them. Click items to collect them — your total must match the target exactly. Too many or too few and you lose a life!"
Conveyor Belt: "Items slide across a belt. Tap the ones you need before they disappear! Your total must match the target."
Help panel content A longer explanation shown when the player clicks the (?) button during gameplay. Includes: a restatement of the goal, a worked example showing a correct move, and a worked example showing an incorrect move. This is the in-game math support — not the Hint Card (which is AI-generated after a loss). "Each round shows a target number. Click items to collect them — your total must match exactly.

✅ Target is 15 → click 8 + 7 = 15 (correct!)
❌ Target is 15 → click 8 + 9 = 17 (too much!)"

Key rule: Engine files must NOT contain any HUD code, victory/game-over screens, sprite loading, font imports, postMessage calls, or Phaser config. All of that lives in the shared base. If you're writing code that isn't about how this specific game option plays, it belongs in the base.

What's hardcoded vs what's AI

This is important to understand: the game logic is built by us, not by AI. AI only handles the creative "skin" on top.

PartWho builds itWhat it means
Game logic (rounds, scoring, win/lose)Hardcoded (us)Every engine has the same structure: 5 rounds, progressive difficulty, randomized numbers. This never changes and always works.
Win conditionHardcoded (us)Win = complete all 5 rounds. Lose = 3 wrong answers in a round. The learner's win condition text (e.g. "complete 5 rounds before time runs out") appears in the intro and help panel, but the actual game logic is the same for all games using that engine.
Game juice (particles, shake, combos)Hardcoded (us)Every game has the same celebration effects, screen shake on mistakes, and combo counter. Built into the shared base template.
Win/lose detectionHardcoded (us)gameWin() and gameLose() are called by the engine code. The parent app listens for these to track mastery, show the Hint Card, and count wins.
Fail trackingHardcoded (us)3 wrong answers in a round = game over. Resets each round. Triggers the Hint Card / Math Moment when the learner loses.
Game title and namesAI (Haiku)The learner's Circuit Board Builder picks are sent to AI, which generates short themed names for the title, character, items, world, and win/lose messages. Takes ~2 seconds.
Background colorsAI (Haiku)AI picks background colors that match the learner's chosen background and character (e.g. deep blue for underwater). Falls back to a default palette if AI fails.
Concept explanationsAI (Sonnet, first time) then cachedThe "What is this?" explanation on each moon is generated once by AI, then cached in Firestore forever. After the first visit, it loads instantly.
Hint Card contentAI (Haiku, first time) then cachedThe math help shown when a learner loses is generated once, then cached. After the first loss on that concept, it loads instantly.
Sprite selectionLearnerLearner picks background, character, and item sprites directly from the library grids in the Circuit Board Builder. No AI involvement — the learner chooses freely. Library built by Mike.
Math round problemsAI (Haiku, per game)For the Phaser engine, AI generates 5 rounds of math problems tailored to the specific Common Core standard. Not generic math — tests the EXACT skill. Built by Mike.

Bottom line: The game itself (how you play, how you win, how you lose) is 100% built by us and always works. The learner picks sprites (background, character, items) and game option. AI generates themed names, colors, and math problems matched to the specific standard.

1. Collect & Manage

Math: Add values to hit a target sum. Shared code: generateValues(target, count) produces item values with a guaranteed valid subset. Color-coding: total turns green at target, red when over. "+value" score pop on collect. Round transition animation.

Game OptionIDGameplayFeelDifficulty Scaling (rounds 1→5)Key Phaser Features

Player can uncollect items (click collected items to return them). Calm, thoughtful. Like picking numbers off a board. More items (5→9). Higher targets (10-20→40-60). More close-value distractors. Fewer obvious subsets — round 5 has several plausible combos that don't quite work. Click-to-collect with fly-to-total tween. Click-to-return for uncollecting. Static item layout with slight random scatter.
Conveyor Belt conveyor-belt A horizontal belt scrolls items right-to-left across the screen. Each item shows its value and moves at a steady pace. Tap/click an item to grab it before it falls off the left edge. Grabbed items stack in a collection tray with a running total. Items that scroll off-screen are gone. Can't un-grab — decisions are permanent. Fast, reactive. Like a factory line where you pick the right parts. Belt speeds up slightly. More items on the belt (more distractors). Higher targets. Items spawn in trickier sequences — the "right" item comes right after a tempting wrong one. Tween-based horizontal scrolling. Item pooling (recycle off-screen). Collision zone at left edge for "missed" items. Collection tray with stacking animation.

More items to distribute (5→9). Both bin targets increase. The split becomes less obvious — round 5 requires careful allocation. Some items could go in either bin (multiple valid solutions exist, but not all placements work). Drag-and-drop with two drop zones. Independent total counters per bin. Item snap-to-bin animation. Visual feedback on each bin (green when target hit, red when over).

12. Learning Science — Research Foundations

These research findings inform how Diagonally teaches math. Every design decision should trace back to one of these principles.

PrincipleWhat the research foundHow we apply itSource
Concrete → Pictorial → Abstract (CPA)Jerome Bruner (1966) established that learners progress through three stages: manipulating physical objects (concrete), seeing visual representations (pictorial), then working with symbols (abstract). A 2025 study confirmed CPA "significantly improved mathematical problem-solving skills" with "deeper understanding and retention." Critical: all three stages should be available simultaneously and learners should move back and forth between them.Moon card shows an animated visual of the concept (pictorial stage). The game itself is the concrete stage (manipulating objects). The hint card provides the abstract stage (formulas, steps). All three available during gameplay.CPA Instruction Study (2025)
CPA Explained
Productive StruggleA 2024 scoping review found that encountering difficulty and working through it (not being told the answer) is "essential for deepening understanding, fostering problem-solving skills, and promoting long-term learning." The learner needs mild cognitive discomfort — enough to engage, not enough to frustrate.First play of a new game is always in hint mode — learner plays their own game WITH the hint card visible. They struggle with the math, use the hints, and learn through doing. Wins don't count until they play "for real."How Productive is the Productive Struggle? (2024)
Supporting Teachers' Understanding (2024)
Intrinsic IntegrationHabgood & Ainsworth (2011) showed that math woven INTO game mechanics produces better learning than math layered ON TOP. A 2024 follow-up found the mechanism is attention — players only attend to features needed for the game task. Math that IS the gameplay gets learned; math sprinkled as quizzes gets ignored.All 19 game engines use intrinsic integration — the math IS the gameplay verb. Balancing an equation IS balancing the scale. Dividing IS cutting the pizza. The learner can't win without doing the math.Habgood & Ainsworth (2011)
Intrinsic Integration & Attention (2024)
InterleavingResearch consistently shows that mixing different types of problems (interleaving) dramatically improves long-term retention compared to practicing one type at a time (blocking). Effective from elementary through college across fractions, algebra, calculus, and geometry.The galaxy is naturally interleaved — learners explore different math domains. The multiplication tables trainer prioritizes weak facts mixed in with strong ones. Playing games across the Library exposes learners to varied concepts.Interleaving in Math — AFT

The Diagonally Learning Flow (research-backed)

  1. Animated visual (3 seconds) — stick figure demonstrates the math concept with actual numbers. One caption sentence. Builds pictorial understanding. (CPA pictorial stage)
  2. Build the game (~30 seconds) — learner picks components in the arcade cabinet. They're invested because it's THEIR game. (Ownership + motivation)
  3. First play with Hint Card — learner plays their own game WITH the hint card visible and math explanation available. They struggle, fail, read hints, try again. Wins don't count. (Productive struggle + CPA concrete/abstract)
  4. Play for real — after practicing, learner plays without hints. Wins count toward demonstration. The math is now the gameplay itself. (Intrinsic integration)
  5. Play others' games — to master, play different games on the same concept built by other learners. Different game options = different ways to apply the same math. (Interleaving)

How Common Core Math is organized

Common Core Math has a 4-level hierarchy:

LevelExampleWhat it isOur galaxy equivalent
DomainNumber & Operations — FractionsBroad math area at one grade levelPlanet (65 total)

Sub-part3.NF.A.3a: "Understand two fractions as equivalent..."A lettered breakdown of a complex standard. Each is a distinct skill.Moon (124 total, included in 517 count)

K-8 is organized by grade: Grade K, Grade 1, ..., Grade 8. Each grade has domains like "Operations & Algebraic Thinking" (OA), "Geometry" (G), "Number & Operations — Fractions" (NF), etc.

High School is organized differently — not by grade but by conceptual category: Number & Quantity (N), Algebra (A), Functions (F), Geometry (G), Statistics & Probability (S). Each category has sub-domains like A-REI (Reasoning with Equations), F-IF (Interpreting Functions), G-CO (Congruence), etc. Our galaxy groups all HS standards under "Grade HS" planets.

Mathematical Practices (MP.1-MP.8) are 8 cross-cutting habits of mind like "Make sense of problems and persevere in solving them." They apply across all grades and are NOT individual skills to demonstrate. They should NOT be moons.

Coherence Map (prerequisite edges)

The Coherence Map from Achieve the Core provides prerequisite relationships between standards. Example: 3.NF.A.1 (understand fractions) is a prerequisite for 5.NF.A.1 (add fractions). Our galaxy uses these as edges — when a prerequisite moon is grey/locked, the dependent moon stays locked. We have 457 prerequisite edges and 84 "related" edges (541 total).
